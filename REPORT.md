# Отчёт по домашней работе №5  
## Оптимизация газа смарт-контрактов (Ethereum, Foundry)

**Дисциплина:** анализ и оптимизация кода смарт-контрактов  
**Репозиторий:** [gas-optimization-quizz](https://github.com/hse-web3sec/gas-optimization-quizz) (вариант с решениями)  
**Инструменты:** Solidity ^0.8.x, Foundry (`forge`, `forge-std`), проверка через `forge test` и `forge test --gas-report`

---

## 1. Цель работы

Получить практические навыки анализа кода смарт-контрактов и снижения стоимости выполнения функций за счёт:

- осознанного использования арифметики, хранения и циклов;
- замены дорогих операций (строковые `require`, лишние чтения storage, линейные проверки) на более дешёвые аналоги;
- сохранения **имён функций** и совместимости с интерфейсами из `Interfaces.sol` и сценариями тестов.

Ограничение из задания: **нельзя считать решением только включение оптимизатора Solidity** — оптимизации заложены в логику и раскладку данных.

---

## 2. Подготовка окружения

1. Установка **Foundry** (в среде WSL Ubuntu: `curl -L https://foundry.paradigm.xyz | bash`, затем `foundryup`).
2. Клонирование репозитория, инициализация зависимостей (`lib/forge-std` через `forge install` / автоматически при первой сборке).
3. Сборка: `forge build`.
4. Тесты: `forge test`.
5. Отчёт по газу (пример):  
   `forge test --match-contract ArrayLength --gas-report`  
   Аналогично для других контрактов из README.

**Замечание по WSL:** при смешивании Windows `PATH` с пробелами (`Program Files`) команда `export PATH=...:$PATH` в bash может падать; для стабильного запуска используется скрипт `scripts/wsl-forge.sh` с «чистым» Linux-`PATH`.

---

## 3. Методика

Для каждой пары «исходный контракт / `*Optimized`»:

- реализован контракт с суффиксом `Optimized` и тестовый контракт `*OptimizedTest` по образцу базовых тестов;
- функции, объявленные в интерфейсах, помечены `override` (требование компилятора Solidity 0.8.8+);
- корректность проверяется полным прогоном: **44 теста, 0 падений** (после исправления конфликта имени параметра с Yul, см. п. 4.9).

## 4. Задания и решения

### Задание 1 — `01. ArithmeticOperators.sol`

| Контракт | Идея оптимизации |
|----------|------------------|
| `AdditionOptimized` | `unchecked { number += value; }` — без лишних проверок переполнения 0.8+ |
| `SubtractionOptimized` | аналогично для вычитания |
| `DivisionOptimized` | деление на 2 и 128 заменено на `>> 1` и `>> 7` (эквивалентно для целых `uint256`) |

**Ожидаемый эффект:** меньше gas на арифметике по сравнению с `/` и проверками overflow.

---

### Задание 2 — `02. ArrayLength.sol`

**Проблема:** в цикле многократно читается `myArray.length`.

**Решение:** вынести длину в локальную переменную `len`, сохранить ту же семантику шага итерации (`i += 2` в `unchecked`), что и в эталонном «антипаттерне».

---

### Задание 3 — `03. CalldataMemory.sol`

**Проблема:** параметр `uint256[] memory` заставляет работать с копией в памяти.

**Решение:** в реализации `CalldataMemoryOptimized` — `uint256[] calldata` (допустимо для `external` при реализации интерфейса), цикл с `unchecked { ++i }`.

---

### Задание 4 — `04. Loops.sol`

**Решение:** инкременты и накопление суммы в `unchecked` блоках во всех трёх вариантах цикла (`for`, `while`, `do-while`), сохранена та же логика границ.

---

### Задание 5 — `05. PackVariables.sol`

**Проблема:** неудачный порядок полей → лишние слоты storage и дорогие `SSTORE` при `setValues`.

**Решение:** упорядочить поля для упаковки в слоты Solidity:

- `uint8 one`, `uint8 six` — в одном слоте;
- `uint256 two` — отдельный слот;
- `bytes18 three` и `bytes14 five` — 18 + 14 = 32 байта в одном слоте;
- `uint8[30] four` — массив в одном слоте (до 32 байт).

---

### Задание 6 — `06. Errors.sol`

**Проблема:** длинная строка в `require(..., "Error: ...")` дорога по развёртыванию и сравнению.

**Решение:** кастомная ошибка `error NotOwner();` и `revert NotOwner()` в модификаторе; поведение для теста (`vm.expectRevert()`) сохранено.

---

### Задание 7 — `07. Swap.sol`

**Решение:** обмен двух `uint256` через XOR (`a ^= b; b ^= a; a ^= b`) вместо трёх сложений/вычитаний.

---

### Задание 8 — `08. ArrayType.sol`

**Решение:** `array = new uint256[](200)` и заполнение по индексу вместо 200 вызовов `push(i)`; инкремент индекса в `unchecked`.

*Примечание:* фактический выигрыш по газу зависит от версии компилятора и того, как учитывается инициализация слотов; сравнение нужно снимать командой `forge test --match-contract ArrayType --gas-report` на вашей машине.

---

### Задание 9 — `09. NestedIf.sol`

**Решение:** одно булево выражение в **inline assembly** (Yul), эквивалентное условию  
`(n > 0) && (n < 100) && (n != 50)`.

**Важно:** параметр нельзя называть `number` — в Yul `number` зарезервировано (номер блока). Использовано имя `n`; сигнатура на уровне ABI совпадает с интерфейсом.

---

### Задание 10 — `10. MultiSigWallet.sol` (`MultiSigWalletOptimized`)

Комплексная оптимизация при сохранении публичного API, событий и строк реверта (`"Not owner"` и др.):

| Область | Мера |
|---------|------|
| Проверка владельца | `mapping(address => bool) ownerLookup` + заполнение в конструкторе; `isOwner` за O(1) вместо цикла по массиву |
| Повторные обращения к storage | локальная ссылка `Transaction storage txn` в `confirmTransaction` / `executeTransaction` |
| Счётчик подтверждений | `unchecked { txn.confirmationCount += 1; }` где безопасно по логике задания |
| `getConfirmations` | подсчёт подтверждений, затем один массив точного размера `new address[](count)` вместо буфера на `owners.length` + второй проход |
| Циклы | `unchecked` для индексов в конструкторе и вспомогательных циклах |

## 7. Выводы

1. Основной выигрыш даёт уменьшение обращений к storage, упаковка слотов, отказ от дорогих revert-сообщений в пользу custom errors, использование `calldata`, `unchecked` там, где инварианты гарантированы, и замена делителей-степеней двойки на сдвиги.
2. Для контракта уровня мультисига критичны структура данных (mapping вместо линейного поиска) и кэширование ссылок на `storage`.
3. Инструменты Foundry позволяют быстро верифицировать и корректность, и потребление газа через `--gas-report`.
4. При разработке под Yul нужно учитывать коллизии имён с встроенными идентификаторами (кейс с `number` в задании 9).
